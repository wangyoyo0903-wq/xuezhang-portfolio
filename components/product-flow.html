1

需求分析阶段

理解用户需求，定义产品方向

▼

📋 核心工作内容

1️⃣ 需求收集
用户反馈、数据分析、竞品调研、业务目标、老板需求
2️⃣ 需求分析

4️⃣ 需求文档输出
BRD商业需求文档、MRD市场需求文档、需求池维护

🎯 真实案例：微信"拍一拍"功能需求分析
📌 需求背景
微信团队发现用户在群聊中有"轻量级互动"的需求。现有的@功能太正式，需要一种更轻松、不打扰的方式引起对方注意。
🔍 用户调研发现
• 群聊中想提醒某人但不想用@（太正式）
• 有时候只是想打个招呼，不需要对方回复
• 现有方案：发表情包、发"在吗"——都显得刻意
📊 竞品分析
• QQ：有"戳一戳"功能，但动画较夸张
• 钉钉：有"DING"功能，偏工作场景
• Slack：有emoji反应，但需要点击选择
📝 需求定义
核心需求：一种轻量级的群聊互动方式
设计原则：
• 操作简单：双击头像即可
• 不打扰：无强提醒，不会震动
• 有趣味：文案可自定义
优先级：P1（用户痛点明确，实现成本低）
📄 需求文档片段
/* 微信拍一拍功能需求文档 */ 功能名称：拍一拍 需求来源：用户调研 + 产品洞察 目标用户：微信全体用户 功能描述： 在群聊和个人聊天中，双击对方头像， 系统提示"你拍了拍XXX"，对方收到 轻量级提示。 交互方式： 1. 双击对方头像 → 触发拍一拍 2. 长按拍一拍提示 → 可设置后缀 验收标准： - 双击响应时间 < 300ms - 不产生强提醒（无震动、无声音） - 支持自定义后缀（最多8个字）
💡 产品经理视角
这个需求的核心洞察是：不是所有互动都需要对方回应。拍一拍填补了"想打招呼但不想打扰"这个空白场景。需求分析时要善于发现用户"没说出口"的需求。
💡 产品经理视角：需求评审会是需求分析阶段的关键节点。会上可能遇到：技术说"做不了"、运营说"没价值"、老板说"优先级不够"。产品经理需要准备好数据支撑、用户调研结果，有理有据地推动需求落地。

🎯 真实案例：淘宝"购物车"功能设计
📌 设计背景
购物车是电商转化的关键环节。淘宝购物车需要承载：商品管理、优惠计算、店铺活动、凑单推荐等复杂功能，同时保持简洁易用。
🎨 原型设计要点
信息架构：
• 店铺分组：同一店铺商品聚合
• 商品信息：图片、标题、规格、价格
• 操作区：数量调整、删除、移入收藏
• 底部栏：全选、合计、结算按钮

交互设计：
• 左滑删除（符合用户习惯）
• 数量调整即时更新价格
• 失效商品单独分组，一键清理
• 优惠信息折叠展示，点击展开
⚠️ 异常流程处理
• 商品下架：显示"商品已下架"，置灰，不可结算
• 库存不足：显示"库存不足"，数量变红
• 价格变动：显示"价格已变动"，提示最新价格
• 优惠券失效：提示"优惠券已失效"，重新选择
• 网络异常：本地缓存，恢复后同步
📄 PRD文档片段
/* 淘宝购物车PRD文档 */ 功能模块：购物车 一、商品展示 1. 按店铺分组，店铺名可点击进入店铺 2. 商品信息：图片(80x80)、标题(最多2行)、 规格、单价、数量 3. 优惠标签：满减、折扣、限时价 二、操作功能 1. 数量调整：+-按钮，最小1，最大库存 2. 删除：左滑删除/编辑模式批量删除 3. 移入收藏：保留商品但不参与结算 4. 选择：单选/全选，未选中不参与结算 三、价格计算 合计金额 = Σ(选中商品单价 × 数量) - 店铺优惠 - 平台优惠 - 优惠券 四、数据埋点 - 购物车曝光PV/UV - 商品点击率 - 删除操作次数 - 结算按钮点击率 - 转化漏斗各环节数据
🔄 评审会反馈与修改
第一轮评审反馈：
• 设计师：店铺活动入口不明显
• 开发：实时价格计算性能有问题
• 运营：缺少凑单推荐入口

修改方案：
• 店铺名下方增加活动标签
• 价格计算改为前端计算+后端校验
• 底部增加"凑单神器"入口
💡 产品经理视角
设计阶段最容易被忽视的是异常流程。新手产品经理只设计"正常情况"，但用户会遇到网络断开、商品下架、库存不足等各种问题。好的设计文档要覆盖80%的异常场景。
💡 产品经理视角：原型评审会可能需要开2-3轮。第一轮被打回很正常，关键是收集反馈后快速修改。评审前要准备好：用户场景、竞品对比、数据支撑，这样更容易说服团队。

⚠️ 真实场景：需求变更怎么办？

场景1：老板临时加需求
开发进行中，老板说"能不能加个分享功能？"
处理方案：评估工作量 → 如果影响上线时间，与老板沟通取舍 → 能延后的放入下一版本 → 必须加的调整排期并通知相关方
场景2：技术实现与预期不符
开发说"这个动画效果实现不了"
处理方案：了解技术限制原因 → 讨论替代方案 → 如果是核心体验，寻找技术突破点 → 如果非核心，接受替代方案
场景3：发现设计遗漏
开发问"网络失败时怎么显示？"
处理方案：快速补充设计 → 更新PRD文档 → 评估是否影响排期 → 记录教训，下次避免

🎯 真实案例：美团外卖"准时达"功能开发
📌 功能背景
用户对外卖配送时间准确性有强烈需求。美团推出"准时达"服务，承诺超时赔付，需要精确预测配送时间。
💻 技术方案评审
前端工作：
• 订单页面显示预计送达时间
• 准时达标签展示
• 超时状态提示

后端工作：
• 配送时间预测算法（机器学习模型）
• 骑手位置实时追踪
• 超时自动赔付逻辑

产品经理关注点：
• 预测准确率要达到多少才能上线？
• 什么情况下不显示准时达？
• 赔付金额如何计算？
📅 排期规划
阶段 时间 负责人
接口设计 3天 后端
前端开发 5天 前端
后端开发 7天 后端
联调测试 3天 全员
🔄 开发过程中的需求变更
变更内容：运营提出需要在高峰期关闭准时达入口
处理方式：
• 评估工作量：增加后台开关配置，约0.5人天
• 影响评估：不影响主流程，可接受
• 决策：同意加入，排期顺延0.5天
• 记录：更新PRD，通知测试补充用例
💡 产品经理视角
开发阶段产品经理要当好"翻译官"和"协调者"。开发看不懂文档时，要用他们能理解的方式解释；遇到技术难点时，要和开发一起讨论替代方案。记住：好的产品经理不是"提需求的人"，而是"帮助团队把需求落地的人"。
💡 产品经理视角：开发阶段最怕"需求蔓延"（Scope Creep）。每个小变更看起来都很简单，但累积起来会严重影响进度。要学会说"不"，或者把新需求放入下一版本。变更要有记录，影响要有评估。

⚠️ 真实场景：产品验收不通过怎么办？

场景1：功能实现与设计不符
验收时发现交互效果和设计稿有差异
处理方案：截图对比 → 明确差异点 → 判断是否影响体验 → 小问题记录后修复，大问题打回重做
场景2：发现严重Bug
验收时发现一个必现的崩溃问题
处理方案：立即阻断上线 → 开发紧急修复 → 重新测试 → 全量回归 → 确认无问题后才能上线
场景3：体验问题
功能没问题，但用起来"感觉不对"
处理方案：具体描述问题（如"动画太快"）→ 与设计确认 → 评估修改成本 → 如果成本低则修改，成本高则记录为优化项

🎯 真实案例：抖音"直播"功能测试验收
📌 测试范围
• 直播推流稳定性
• 观众端播放流畅度
• 礼物、弹幕互动功能
• 多机型兼容性
• 弱网环境表现
📝 测试用例示例
用例ID 测试场景 预期结果 优先级
TC001 正常开播 推流成功，观众可观看 P0
TC002 弱网开播 提示网络不佳，自动降画质 P0
TC003 发送礼物 主播收到，观众看到动画 P1
TC004 断网重连 自动重连，不退出直播间 P0
🐛 发现的典型Bug
Bug1：礼物动画卡顿
• 现象：连续发送多个礼物时，动画卡顿
• 原因：动画资源未预加载
• 修复：增加预加载机制

Bug2：弱网下崩溃
• 现象：网络切换时App崩溃
• 原因：网络状态监听空指针
• 修复：增加空判断

Bug3：弹幕重叠
• 现象：弹幕量大时文字重叠
• 原因：弹幕轨道计算错误
• 修复：优化轨道分配算法
✅ 产品验收要点
• 直播画质是否清晰流畅
• 礼物动画是否精美
• 弹幕滚动是否自然
• 弱网提示是否友好
• 整体体验是否符合预期

验收结果：发现3个体验问题，修复后通过验收
💡 产品经理视角
测试阶段是产品经理的"守门员"时刻。要站在用户角度验收产品，不要只测试"正常情况"，要专门测试"用户会怎么乱操作"。发现的问题要及时记录、跟踪修复进度。
💡 产品经理视角：产品验收时不要怕"找茬"。现在发现问题总比上线后被用户发现好。验收时要模拟真实用户场景，用"小白"的心态去操作，而不是用"专家"的思维。

⚠️ 真实场景：上线出问题怎么办？

场景1：灰度期间发现严重Bug
灰度5%时发现支付功能异常
处理方案：立即停止灰度 → 定位问题 → 紧急修复 → 重新测试 → 重新灰度
场景2：应用市场审核被拒
App Store审核说违反了某条规定
处理方案：仔细阅读拒绝原因 → 修改问题 → 重新提交 → 必要时申诉沟通
场景3：上线后用户投诉
大量用户反馈新版本不好用
处理方案：收集具体反馈 → 分析问题原因 → 如果严重则回滚版本 → 如果不严重则快速迭代修复

🎯 真实案例：支付宝"集五福"活动上线
📌 活动背景
春节集五福是支付宝年度S级活动，涉及数亿用户参与，对系统稳定性要求极高。
📋 上线清单（部分）
检查项 负责人 状态
功能测试通过 测试组 ✅
性能压测通过 运维组 ✅
安全审计通过 安全组 ✅
应急预案准备 产品组 ✅
客服培训完成 运营组 ✅
🚀 灰度发布策略
Day 1：内部员工测试（100人）
Day 2：灰度1%（约100万用户）
Day 3：灰度5%（约500万用户）
Day 4：灰度20%（约2000万用户）
Day 5：全量发布

灰度期间监控指标：
• 服务器CPU/内存使用率
• 接口响应时间
• 错误率
• 用户反馈量
⚠️ 灰度期间遇到的问题
问题：灰度5%时，部分用户反馈"福卡加载慢"

排查：
• CDN节点缓存未命中
• 图片资源过大

解决：
• 紧急扩容CDN节点
• 压缩图片资源
• 继续观察，确认问题解决后扩大灰度
💡 产品经理视角
大型活动上线要宁可慢，不要错。灰度发布是发现问题的最后一道防线。灰度期间要密切监控数据，任何异常都要停下来排查。同时要准备好回滚方案，万一出问题能快速恢复。
💡 产品经理视角：上线不是终点，而是新的起点。上线后要密切关注用户反馈和数据表现，准备好应急预案。如果出现严重问题，要知道如何快速回滚或发布修复版本。记住：上线只是开始，运营才是长跑。
6

运营迭代阶段

持续优化产品，提升用户体验

▼

📋 核心工作内容

1️⃣ 数据监控与分析
用户行为分析、功能使用数据、转化漏斗、留存分析
2️⃣ 用户反馈收集

⚠️ 真实场景：数据不达预期怎么办？

场景1：新功能使用率低
上线的新功能，用户使用率只有5%
处理方案：分析原因（入口太深？价值不清晰？）→ 优化入口位置 → 增加引导提示 → 观察数据变化 → 如果仍无改善则考虑下线
场景2：用户流失增加
新版本上线后，次日留存下降5%
处理方案：定位流失用户群体 → 分析流失原因 → 是否新版本体验问题 → 如果是则快速修复 → 如果不是则分析竞品动态
场景3：转化率下降
购买转化率比上线前低了10%
处理方案：分析漏斗各环节 → 找到流失增加的环节 → 定位原因（流程变长？页面加载慢？）→ 针对性优化

💡 产品经理视角
产品是"长"出来的，不是"做"出来的。上线后要根据数据和反馈持续优化。建立数据监控体系，定期复盘，形成"上线-观察-优化-再上线"的良性循环。记住：没有完美的产品，只有不断进化的产品。
💡 产品经理视角：运营迭代阶段要建立"数据驱动"的习惯。不要凭感觉做决策，要用数据说话。同时要平衡"用户想要的"和"产品应该做的"，有时候用户反馈只是表象，真正的问题需要深入分析。
